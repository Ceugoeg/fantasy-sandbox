# Q
C++ 的模板是编译期决定的。如果一个实体在编译时被定义为 `Entity<RandomWalkPolicy, PhysicsPolicy>`，那么在游戏运行期间，LLM 是绝对不可能凭空给它塞入一个 `AIPolicy` 的（因为那会改变实体的内存布局和类型签名，需要重新编译）。我们如何用编译期的 Entity.h 实现运行时的 LLM 控制？

# A
这个问题问到了整个引擎架构最核心的灵魂所在。在用 C++ 这种静态编译型语言构建“大模型运行时模组 (Runtime Modding)”生态时，我们面临的最大敌人就是**编译期锁死**。

答案正是我们即将在 Phase 1.4 构建的 Event Bus (事件总线)。

我们可以采用 **“动静结合”** 的混合架构：

1. 静态基石（性能担当）：像碰撞检测、基础寻路、随机游荡这些需要每秒执行 20 次的极高频逻辑，依然作为 C++ Policy 写入模板参数，享受极致的内联优化。

2. 动态外挂（LLM 接口）：我们编写一个特殊的 `EventDrivenPolicy` 或 `Lua/LLMScriptPolicy`，并在编译时塞进所有想要被大模型控制的实体中。这个 Policy 不包含写死的逻辑，而是负责监听 Event Bus 上的特定指令，并将实体的状态改变广播出去。大模型只需往 Event Bus 里扔 JSON 指令，实体内的 `EventDrivenPolicy` 就会捕获并执行。

这样一来，我们既保住了 C++ 底层的极致性能，又为 LLM 留出了运行时的动态操控后门。

为了给你详尽的、原理层面的解释，我们把这个“动态外挂”架构拆解为**底层机制**和**数据流向**两个维度来看。

### 一、 原理剖析：静态装甲与动态神经

在 C++ 中，由于泛型（Template）和内存布局在编译那一刻就已经焊死，我们绝对无法在游戏跑起来之后，强行给一个只懂 `RandomWalkPolicy`（随机游荡）的史莱姆塞入一段由 LLM 实时生成的 C++ 代码。

因此，所谓的“动态外挂”，在底层原理上其实是**数据驱动 (Data-Driven)** 与 **事件监听 (Event-Listening)** 的结合。我们可以将其称为 **`LLMProxyPolicy`（大模型代理组件）**。

#### 1. 编译期的“留后门”
在构建实体时，我们利用刚刚敲定的变长模板机制，给那些打算让大模型控制的实体“预装”一个特殊的策略组件：
```cpp
// 编译期焊死的装甲：基础物理 + 随机游荡 + LLM 代理监听器
using SmartSlime = Entity<PhysicsPolicy, RandomWalkPolicy, LLMProxyPolicy>;
```
在这个组合中，`PhysicsPolicy` 负责硬核的碰撞和重力（每秒高频运算，榨干 CPU），而 `LLMProxyPolicy` 本身**不包含任何具体的行为逻辑**。它只是一个空壳子，或者说是一个“接收器”。

#### 2. 运行时的“提线木偶”机制 (The Puppet Mechanism)
`LLMProxyPolicy` 的核心原理是：**持有状态，并监听总线。**
* **注册耳朵**：当史莱姆被创建时，它的 `LLMProxyPolicy` 会向全局的 Event Bus 订阅一个名为 `LLMCommandEvent` 的事件，并过滤出针对自己 ID 的指令。
* **接收神谕**：当大模型决定干预世界时，它输出一段 JSON。Node.js 捕获后，通过标准输入（stdin）把 JSON 塞给 C++ 引擎。C++ 的主循环解析这个 JSON，并把它打包成一个 `LLMCommandEvent` 扔到 Event Bus 上。
* **状态覆写 (Override)**：史莱姆的 `LLMProxyPolicy` 听到了这个事件，比如指令是“向坐标 (10, 20) 移动”。此时，该策略会在内部修改实体的目标坐标状态，并在接下来的 `tick()` 循环中，**强行覆盖或阻断**原本 `RandomWalkPolicy` 的漫无目的行为，转而执行 LLM 赋予的确切目标。

**一句话总结原理**：大模型并没有在运行时写 C++ 代码，它写的是高维度的 JSON“意图”。C++ 引擎在编译期就已经埋好了能够解析这些意图的“受体”（Proxy Policy），在运行时通过 Event Bus 实现对实体行为的动态劫持。

### 二、 这一步应该在什么阶段实现？

在软件工程规范中，顶层建筑必须依赖坚实的底层基础设施。实现这个机制有着严格的前置依赖：

1.  **必须先有泛型载体**：你需要支持变长模板的 `Entity.h`，才能把这个后门挂件和常规物理挂件拼在一起。（这是我们 **Phase 1.4** 正在做的事）
2.  **必须先有神经中枢**：你需要一个能传递 `std::any` 的 `EventBus`，才能让指令在系统间无损流转。（这依然是 **Phase 1.4** 正在做的事）
3.  **必须先有输入管道**：引擎目前是“聋子”，你需要完成 `stdin` 的监听与解析。（这是接下来的 **Phase 1.5: Input Oracle**）

**结论**：
真正的 `LLMProxyPolicy`（动态外挂组件）的编写与接入，应当被放在 **Phase 2（法则注入）** 的初期，或者至少要在 **Phase 1.5（输入枢纽）** 完全打通并测试完毕之后再进行。

现在，我们正处于打地基的关键时刻。有了这些明确的原理认知，我们就可以确保当前写的 Event Bus 和变长模板 `Entity.h` 是完全朝着支撑这个宏大目标去的，不会走弯路。
