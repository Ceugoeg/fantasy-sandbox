# Q
既然 `BaseEntity` 已经能作为实体的躯壳，为什么还要在中间卡一个 `Entity.h`（也就是 `Entity<Policy...>`）？

# A
答案藏在 **C++ 强类型系统的容器限制** 和 **虚函数（Virtual Function）的性能开销** 里。我们来深入到底层原理：

### 1. `BaseEntity` 的宿命：为了被统一收纳 (Type Erasure)

在 `World` 引擎的核心循环里，我们需要一个列表来管理世界上的所有东西，遍历它们并调用更新逻辑。
```cpp
// 在 World.h 中，我们必须这样写：
std::vector<std::unique_ptr<BaseEntity>> all_entities;
```
为什么容器的类型必须是 `BaseEntity`？因为 C++ 的 `std::vector` 是**同质容器（Homogeneous Container）**，它里面只能装完全一样类型的东西。

如果你的史莱姆是由模板拼装出来的：
* 瞎逛的史莱姆类型是：`Entity<RandomWalkPolicy>`
* 追踪玩家的史莱姆类型是：`Entity<FollowPlayerPolicy>`

在 C++ 编译器眼里，**这是两个完全不同的类**，就像 `int` 和 `string` 一样风马牛不相及。你绝对不可能把它们塞进同一个 `std::vector<Entity>` 里，因为根本不存在一个独立的叫 `Entity` 的类型，它只是个模板蓝图。

所以，我们需要 `BaseEntity` 作为一个**没有任何模板参数的非泛型基类**，来抹除它们在类型上的差异（Type Erasure），让 `World` 能用一个统一的指针数组把它们全装进去。它负责存放所有实体共有的纯数据：`id`、`x`、`y`、`state`。

### 2. `Entity.h` 的使命：无开销的“装甲拼装台”

既然 `World` 只认识 `BaseEntity`，它在每帧循环（Tick）时怎么触发那些特殊的逻辑（Policy）呢？传统的做法是给 `BaseEntity` 写一个 `virtual void update() = 0;`。但虚函数会带来动态绑定的开销（查虚函数表 vtable），在游戏引擎底层遍历成千上万个实体时，这是致命的性能杀手。

**`Entity.h` 就是为了解决这个问题而诞生的。** 它是连接“纯数据基类”和“各种行为零件”的模板外骨骼：

```cpp
// 概念演示：Entity.h 内部的样子
template <typename... Policies>
class Entity : public BaseEntity, public Policies... {
public:
    // 构造函数：初始化数据，并将各种 Policy 注入
    Entity(int id, float start_x, float start_y) 
        : BaseEntity(id, start_x, start_y) {}

    // 真正的逻辑入口
    void tick(float dt) override {
        // C++17 魔法：折叠表达式 (Fold Expression)
        // 在编译期，它会直接把所有 Policy 的逻辑内联（Inline）展开在这里
        // 完全没有虚函数调用的开销！
        (this->Policies::execute(this, dt), ...); 
    }
};
```

### 总结：经典的底层架构分工

* **`BaseEntity`**：对上（World）负责。它是统一的接口和数据容器，保证所有实体能在内存中被连续的指针数组安全管理。
* **`Entity.h`**：对下（Policy）负责。它是一个编译期的组装车间，把不同的行为策略像拼积木一样完美无缝地焊死在实体上，榨干 CPU 的每一滴性能。

所以，`BaseEntity` 和 `Entity<T>` 缺一不可。前者骗过了 C++ 严苛的运行时容器检查，后者利用了 C++ 强大的编译期模板内联特性。
