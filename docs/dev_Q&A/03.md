# Q
在引入 Tiled 导出的带有旋转和翻转状态的图块后，原本完美的 Canvas 网格渲染出现了严重的视觉 Bug。最初表现为随缩放变化的黑色网格缝隙（拓扑撕裂），在尝试修复后又演变成了夹杂相邻图块颜色的绿色/白色细线（纹理出血）。
导致这些现象的底层数学原理是什么？我们是如何经过三次迭代，最终彻底解决它的？

# A
这次排错过程触及了 2D 游戏引擎渲染管线中最核心的三个陷阱：**浮点数取整悖论**、**IEEE 754 标准下无理数的精度丢失**，以及 **GPU 亚像素双线性插值采样**。

我们在解析 Tiled 地图时，遇到了通过高位掩码（0x80000000, 0x40000000, 0x20000000）记录的旋转与翻转数据。为了在 Canvas 中还原这些状态，我们的渲染器经历了以下三个阶段的演进：

### 尝试一：四舍五入导致的“拓扑撕裂” (Topological Tearing)
* **我们的做法**：为了追求像素对齐，我们使用了 `Math.round` 对旋转中心点和绘制宽高进行了强制取整。
* **失败现象**：地图上出现了黑色的网格缝隙，且只有旋转过的图块边缘会产生不对称的粗细变化。
* **底层死因**：这是一个经典的数学悖论——**各部分取整的和，不等于总和的取整**（即 Round(A) + Round(B) != Round(A+B)）。当发生 90 度旋转时，原本的 X 轴偏移量和 Y 轴偏移量发生了互换。由于之前对浮点数中心点（如 24.5）进行了 `Math.round` 强制加法，导致旋转图块的坐标系与未旋转图块的坐标系产生了 1 像素的非对称错位，原本严丝合缝的网格被撕裂，露出了底部的黑色画布。

### 尝试二：浮点数 $\pi$ 与双线性插值的灾难 (Sub-pixel Bleeding)
* **我们的做法**：为了修复坐标错位，我们移除了 `Math.round`，保留了带有 `.5` 的浮点绝对精度让矩阵自己去抵消。同时为了弥合视觉缝隙，对绘制宽高强制增加了 `+0.5` 的像素补偿，并使用常规的 `ctx.rotate(Math.PI / 2)` 执行旋转。
* **失败现象**：黑色缝隙消失，取而代之的是所有图块边缘出现了绿色、白色的细线，即著名的**纹理出血 (Texture Bleeding)**。
* **底层死因**：计算机底层的 IEEE 754 浮点数无法精确表示无理数 $\pi$。当我们调用 `Math.PI / 2` 时，底层算出的其实是一个极度接近 90 度但带有微小尾数的值（如 cos 值不为 0 而是 6.12323e-17）。
这导致图块在显存中发生了不到亿分之一度的微小倾斜！加上我们为了容错加入的 `+0.5` 拉伸，直接强行激活了 GPU 的**双线性插值采样器 (Bilinear Filter)**。它在读取紧凑排列的 16x16 图集 (Atlas) 时，跨越了物理边界，把相邻的草地和花朵像素错误地“吸”了进来，形成了那些绿白相间的幽灵线条。

### 最终方案：纯整数仿射矩阵劫持 (Integer Affine Matrix)
* **解决方案**：彻底抛弃 `ctx.rotate()`、`ctx.translate()` 等带有浮点三角函数运算的上层封装，直接接管底层的数学仿射变换矩阵。
* **成功原因**：对于 Tiled 协议中所有的 90 度级旋转与镜像翻转，其对应的矩阵 $\begin{bmatrix} a & c & e \\ b & d & f \end{bmatrix}$ 参数必定是纯粹的 1、0 或 -1。没有任何小数，没有任何 $\pi$。
我们通过计算，直接向 Canvas 注入 `ctx.transform(a, b, c, d, tx, ty)`。因为所有操作数全是绝对整数，底层的光栅化引擎（Rasterizer）进入了完美的“像素级一对一映射”模式，彻底禁用了抗锯齿和次像素插值。无论怎么缩放，渲染结果都如同硅晶片般严丝合缝，再无任何精度溢出与纹理出血。
